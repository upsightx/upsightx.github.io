["Conventional Commits](https://www.conventionalcommits.org/) specification:\n\n```\n<type>(<scope>): <subject>\n\n<body>\n\n<footer>\n```\n\n## Type Categories\n\n- `feat`: New feature\n- `fix`: Bug fix\n- `docs`: Documentation changes\n- `style`: Code style changes (formatting, etc.)\n- `refactor`: Code refactoring\n- `perf`: Performance improvements\n- `test`: Adding or updating tests\n- `chore`: Maintenance tasks\n- `ci`: CI/CD changes\n\n## Subject Rules\n\n- Use imperative mood (e.g., \"Add\" not \"Added\" or \"Adds", "Use English language\n- Don't end with a period\n- Keep it short (< 72 characters)\n- Capitalize first letter\n\n## Examples\n\n### Good\n```\nfeat: Add behavior-learner skill for user behavior analysis\ndocs: Update MEMORY.md with model-manager completion\nchore: Update behavior data with 5th heartbeat record\n```\n\n### Bad\n```\nfeat: \u6dfb\u52a0\u884c\u4e3a\u5b66\u4e60\u5668 (Chinese)\nAdd behavior-learner (missing type)\nfeat add behavior-learner (no space after :)\n```\n\n## Additional Notes\n\n- All commit messages should be in **English** for consistency\n- Use clear, concise descriptions\n- Group related changes into logical commits\n- Avoid mixing multiple features in one commit\n\n## Example Workflow\n\n```bash\n# Make changes\ngit add .\n# Commit with proper format\ngit commit -m \"feat: Add new feature description", "Push\ngit push origin main\n```"]
